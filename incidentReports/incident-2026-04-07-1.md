# Incident: 2026-04-04 17-07-13

## Summary

> Temporary outage at Pizza Factory! For 2 hours 3 minutes 27 seconds on April 4th, 2026, all pizza orders were unfulfilled during this time period. The error stemmed from planned chaos testing. No end users were impacted.

```md
From 17:07:13 UTC to 19:10:40 UTC on April 4th 2026, 4 test users encountered failed pizza orders. The event was triggered by intentional chaos testing arranged by the corporate office in order to test response times. The Recovery Time Objective (RTO) set by corporate policy is a window of 4 hours.

The chaos test temporarily disabled all API calls to the pizza factory until the appropriate log file was found. The event was detected by a manual check of the metrics by the team member on call. The team member checked the logs and made the necessary API call to the corporate pizza factory acknowledging that the chaos had been detected and discovered. This moderate incident affected 4 test users.
```

## Detection

```md
This incident was not detected by the automated system and so the team member on call was not notified. Fortunately the team member on call decided to manually check the metrics dashboard during his lunch break. Discovery occurred at 18:38:53 UTC. The team member began work immediately to diagnose and rectify the issue.

This was due to a shortage of alerts on base metrics as the DEVOPS team was expecting the chaos test to come through a specific API for the purpose. The DEVOPS team learned from this oversight and will expect future chaos test to target critical systems.
```

## Impact

```md
For 2 hours 3 minutes 27 seconds between 17:07:13 UTC and 19:10:40 UTC on 2026-04-04, all pizza orders were unfulfilled.

This incident affected 0 customers and 4 test users, who suffered from failed orders. No accounts were billed in error.

Due to the small clientele of the SLC franchise, no customers were affected and no reports were made.
```

## Response

```md
After noticing the issue at 18:38:53 UTC, the on-call team member inspected the logs, confirmed that the outage matched the planned chaos test, and used the corporate support endpoint to acknowledge and end the exercise.

No escalation was required because the failure was isolated to the test flow, but the manual discovery added roughly 91 minutes of avoidable delay.
```

## Root cause

```md
A planned chaos test disabled the pizza factory order API path, and alerting was wired to a separate signal instead of the customer-facing fulfillment path. When the simulated traffic source went idle, the system stopped generating the conditions needed to trigger the expected alert, so the incident stayed hidden until manual inspection.
```

## Resolution

```md
The on-call team validated the failure in logs, used the follow-link-to-end-chaos support action to restore normal API behavior, and confirmed that new pizza orders were flowing again.

The incident was closed once fulfillment recovered and no additional failed orders were observed.
```

## Prevention

```md
This same monitoring gap could affect any future chaos exercise or real outage that suppresses the order API without tripping the expected alert.

The core issue is that detection depends on a specific test path instead of the actual fulfillment path.
```
